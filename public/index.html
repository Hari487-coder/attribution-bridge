Manual distribute

The live pipeline does the same automatically via the webhook.

    

      

        
        
      

      

        
        
      

    

    
    

  

  

    
Channel test

    
Creates a disposable test contact through the configured token, reads back the createdBy.source stamp GHL actually assigned, then deletes it. Run this once per broker before trusting the pipeline — it answers whether this token type earns the INTEGRATION stamp.

    
    
    
    

    

  







  

    
Bulk import: master → broker

    
Pull existing contacts from your master account and create them in a broker sub-account through the API channel — for onboarding a new broker or backfilling leads that arrived before the webhook was wired. Each contact runs through the same verify-first pipeline: only genuinely opted-in leads are pushed, opt-outs are refused, and duplicates follow your duplicate policy. Dry-run first.

    
    
    
    

    

      
      
      
      
      
    

    

    

      
      

        
        
      

    

    

  







  

    
Verified attribution registry

    
The bridge refuses to distribute a lead unless its master record shows genuine opt-in (attribution or a real Meta/IG integration) — so a cold contact dumped into the master can't be laundered past DNC. Each verification is signed (HMAC) and workspace-scoped. An opt-out (withdraw) always wins and is never overridden.

    
    

  

  

    
Manually verify a master contact

    
Fetches the contact from the master location, checks opt-in evidence, and registers it if it passes. Use for leads that arrived before the webhook was wired.

    
    
    
    

  

  

    
Record an opt-out

    
Withdraws a number. Future bridges of this number are refused. Wire GHL's opt-out/STOP workflow to POST /webhook/optout to automate this.

    
    
    
    
    
    

  







  

    
Backlog migration

    
Existing copies made with GHL "Copy Contact" keep their old stamp forever (createdBy is immutable). This scans a broker location for contacts that would fail the evidence check, then deletes + recreates them through the API channel. Dry-run first, always.

Why this works

    
CastigliaAI allows an outbound AI call to a national-DNC number only when the GHL contact shows opt-in evidence: populated attributionSource, populated lastAttributionSource, or createdBy.source = "INTEGRATION". GHL's native "Copy Contact" produces none of these. Contacts created via an API integration get the INTEGRATION stamp — so this app replaces Copy Contact with an API create.

Hosting the webhook

    
GHL must reach this app over HTTPS. Quick options: cloudflared tunnel --url http://localhost:3344 for a persistent tunnel from this machine, or deploy the folder to Render/Railway (it's a single node server.js, no build step). Set a dashboard password before exposing it.

  

  

    
How the verification model works

    
    
      
  • Verify-first: the bridge checks the master contact's genuine opt-in (attribution / real Meta integration / not-DND) — fields GHL sets and the customer can't forge — and refuses to bridge anything that fails. A cold contact dumped into the master won't pass.
    • 
      
  • Signed & workspace-scoped: each verification is HMAC-signed and stored per phone, defensible across this master's brokers but never shared to unrelated workspaces.
    • 
      
  • Opt-out always wins: a withdrawal is never overridden by a later verification. Wire GHL's STOP/opt-out workflow to /webhook/optout.
    • 
      
  • Honest about enforcement: CastigliaAI does not read the signature — it still allows the call because of the INTEGRATION stamp the bridge applies. The security gain is the webapp refusing to stamp non-opted-in contacts. The signed note is an audit trail + a forward-compat hook if CastigliaAI later adds real server-side verification.
    • 
      
  • createdBy is immutable → backlog migration deletes + recreates, which drops that contact's GHL history. Migrate only un-worked contacts. Standard fields, tags, and fieldKey-matched custom fields are preserved; conversations/tasks are not.
